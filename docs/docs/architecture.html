Architecture

How CodeSight is structured internally.

Module Overview

Project Structure
codesight/
├── __init__.py          # version, metadata
├── __main__.py          # python -m codesight entry point
├── cli.py               # argparse CLI, output formatting
├── config.py            # AppConfig, ProviderConfig, load/save
├── analyzer.py          # core analysis engine, system prompts
├── compression.py       # code map builder, token reduction
├── streaming.py         # streaming output for all providers
├── templates.py         # custom prompt template management
├── cost.py              # token cost estimation per model
├── pipeline.py          # multi-model triage → verify pipeline
├── benchmark.py         # LLM benchmark on vulnerable samples
├── sarif.py             # SARIF output generation
└── providers/
    ├── base.py          # BaseLLMProvider, Message, LLMResponse
    ├── factory.py       # provider registry and instantiation
    ├── openai_provider.py
    ├── anthropic_provider.py
    ├── google_provider.py
    └── ollama_provider.py

Data Flow

Every analysis follows the same path:

  1. CLI parses the command and loads config
  2. Analyzer reads the source file and validates size
  3. Compression builds a code map if the file exceeds 300 lines
  4. System prompt is selected based on the task (review/bugs/security/etc.)
  5. Provider sends the prompt + code to the LLM API
  6. Response is parsed, formatted (markdown/json/sarif), and printed
  7. Cost is estimated from token usage

Provider Abstraction

All providers implement BaseLLMProvider:

providers/base.py
class BaseLLMProvider(ABC):
    async def complete(messages, max_tokens, temperature) -> LLMResponse
    async def health_check() -> bool
    def name -> str

Adding a new provider means implementing these three methods and registering it in factory.py.

Compression

For files over 300 lines, the compression module extracts:

  • Import statements
  • Function/class/method signatures
  • Structural outline with line numbers

This typically reduces token usage by 60-80% while preserving enough context for accurate analysis.

Multi-Model Pipeline

The pipeline feature chains two models for security analysis:

  1. Triage — a fast/cheap model (e.g., Llama 3 via Ollama) scans the code and flags suspicious areas
  2. Verify — a strong model (e.g., GPT-5.4) does deep analysis only on flagged sections

This reduces cost by 70-85% compared to running the full code through the expensive model, with minimal accuracy loss.

Configuration Hierarchy

  1. CLI flags (--provider, --output)
  2. Environment variables (OPENAI_API_KEY, CODESIGHT_MODEL)
  3. Config file (~/.codesight/config.json)
  4. Defaults (OpenAI, GPT-5.4, markdown output)
